## 四、重要缺失功能 (按优先级排序)

### P0: 核心功能缺失

#### 1. 响应式压缩 (Reactive Compaction)
- **现状**: 仅有主动压缩 (在请求前检查 token 使用率)
- **缺失**: 无法处理 API 返回的 `prompt-too-long` 错误
- **影响**: 长对话可能因为估算不准确而导致 API 错误无法恢复
- **原版实现**: 4 种压缩策略——常规、微压缩、上下文折叠、裁剪，外加响应式压缩

#### 2. 模型降级 (Model Fallback)
- **现状**: 完全缺失
- **缺失**: 无法在模型过载 (529) 时自动切换到备用模型
- **影响**: 高负载时服务不可用
- **原版实现**: 连续 529 后触发 FallbackTriggeredError，切换模型重试

#### 3. 流式工具执行 (Streaming Tool Execution)
- **现状**: 工具在完整响应收集后才执行
- **缺失**: 无法在 API 流式响应中提前执行工具
- **影响**: 多工具调用时的延迟增加
- **原版实现**: StreamingToolExecutor 在工具块流入时即开始执行，支持并发控制

#### 4. 工具搜索/延迟加载 (Tool Search/Deferral)
- **现状**: 完全缺失
- **缺失**: 所有工具都在初始请求中发送
- **影响**: prompt 空间浪费，尤其 MCP 工具多时
- **原版实现**: shouldDefer 标记 + ToolSearchTool 按需加载

### P1: 重要功能缺失

#### 5. 会话持久化格式
- **现状**: JSON 完整快照
- **缺失**: JSONL 追加日志，增量写入，崩溃恢复
- **影响**: 大会话保存开销高，崩溃丢失最后一轮

#### 6. 成本追踪
- **现状**: 硬编码费率 ($3/M input, $15/M output)
- **缺失**: 分模型定价、缓存 token 差异化定价、预算限制 (maxBudgetUsd)
- **影响**: 成本显示不准确

#### 7. 信任对话框 (Trust Dialog)
- **现状**: 完全缺失
- **缺失**: 首次在新目录运行时的安全确认
- **影响**: 安全隐患——恶意 .claude/settings.json 可通过 apiKeyHelper 执行任意命令

#### 8. OAuth 认证
- **现状**: 仅支持 API key 和 Bearer token
- **缺失**: OAuth 流程、Keychain 集成、token 刷新
- **影响**: 无法通过 claude.ai 账号认证

#### 9. Git Worktree 工具
- **现状**: 完全缺失
- **缺失**: EnterWorktree/ExitWorktree 工具
- **影响**: 无法创建隔离的 git worktree 进行开发

#### 10. 设置迁移系统
- **现状**: 完全缺失，仅依赖 `#[serde(default)]`
- **缺失**: 版本化迁移脚本
- **影响**: 设置格式变更时无法自动迁移

### P2: 增强功能缺失

#### 11. MCP SSE/HTTP/WebSocket 传输
- **现状**: 仅实现 stdio 传输
- **缺失**: SSE, HTTP, WebSocket 传输
- **影响**: 无法连接远程 MCP 服务器

#### 12. CLAUDE.md @include 指令
- **现状**: 完全缺失

- **影响**: 无法在 CLAUDE.md 中引用其他文件

#### 13. 团队/Swarm 协作
- **现状**: AgentTool 支持子代理，但无跨进程协作
- **缺失**: 团队创建/消息传递/协调器模式
- **影响**: 无法使用多代理并行工作

#### 14. Vim 模式
- **现状**: 完全缺失
- **影响**: 熟悉 vim 的用户无法使用键绑定

#### 15. 语音模式

#### 16. 远程控制 (Bridge/CCR)
- **现状**: 完全缺失
- **缺失**: 远程会话管理、WebSocket 双向通信

## 五、已知问题和改进建议

### 架构问题

#### 1. 会话存储格式不当
- **问题**: 使用完整 JSON 快照存储会话，每次保存都序列化整个会话
- **建议**: 迁移到 JSONL 追加日志格式，只追加新消息，减少 I/O 开销
- **参考**: 原版使用 `BufferedWriter` + 100ms 延迟序列化

#### 2. 压缩策略过于单一
- **问题**: 只有一种基于 LLM 摘要的压缩方式
- **建议**: 增加微压缩 (清理工具结果) 和响应式压缩 (处理 API 错误)
- **参考**: 原版有 4 种策略，根据场景选择

#### 3. 工具执行时序
- **问题**: 等待完整 API 响应后才执行工具
- **建议**: 实现流式工具执行器，在工具块完整后立即执行
- **参考**: 原版 StreamingToolExecutor

### 安全问题

#### 4. 缺少信任对话框
- **问题**: 在不受信任的目录中运行时，.claude/settings.json 中的 apiKeyHelper 可执行任意命令
- **建议**: 首次运行时要求用户确认信任
- **严重级**: 高

#### 5. 缺少沙箱
- **问题**: Bash 工具直接执行命令，无隔离
- **建议**: 至少实现文件系统路径限制
- **严重级**: 中

### 性能问题

#### 6. 缺少文件状态缓存
- **问题**: 每次 FileEdit/FileWrite 不检查文件是否被修改
- **建议**: 实现 LRU 文件状态缓存，跟踪读取时间戳
- **参考**: 原版 FileStateCache (100 entries, 25MB)

#### 7. Prompt 空间浪费
- **问题**: 所有工具 schema 都在初始请求中发送
- **建议**: 实现工具延迟加载，按需发送 schema
- **参考**: 原版 shouldDefer + ToolSearchTool

## 十、总结

rust-claude-code 已经是一个**功能可用、架构清晰**的 Rust 重实现，覆盖了原版约 **55%** 的功能。核心 agentic loop、权限系统、流式传输、工具执行、TUI 渲染等关键路径已经完善。

主要差距集中在:
1. **韧性** (响应式压缩、模型降级、崩溃恢复)
2. **性能** (流式工具执行、文件状态缓存、工具延迟加载)
3. **安全** (信任对话框、沙箱)
4. **企业级功能** (OAuth、功能开关、分析遥测、远程控制)

项目的代码质量和测试覆盖率很高，架构设计支持后续功能扩展。最紧迫的工作是补齐 P0 级韧性和安全功能。

### 迭代计划中未覆盖但原版有的重要功能

以下功能既不在迭代计划中，也未实现，但原版 Claude Code 具备:

| 功能 | 说明 | 影响 |
|------|------|------|
| 响应式压缩 | 对 API prompt-too-long 错误的压缩恢复 | 长对话可能崩溃 |
| 模型降级 | 529 过载时自动切换备用模型 | 高负载不可用 |
| 流式工具执行 | 在 API 流式响应中提前执行工具 | 多工具延迟高 |
| 工具延迟加载 | shouldDefer + ToolSearchTool | prompt 空间浪费 |
| 信任对话框 | 首次运行的安全确认 | 安全隐患 |
| OAuth 认证 | claude.ai 账号登录流程 | 认证方式有限 |
| JSONL 会话格式 | 追加日志 + 崩溃恢复 | 大会话性能差 |
| 文件状态缓存 | LRU 缓存跟踪文件读取 | 无过期检测 |
| 设置迁移系统 | 版本化迁移脚本 | 格式变更需手动处理 |
| 功能开关 | 编译时/运行时功能开关 | 无法 A/B 测试 |
| @include 指令 | CLAUDE.md 文件引用 | 无法模块化指令 |
