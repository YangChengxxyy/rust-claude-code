## Context

当前 Rust Claude Code 的 TUI 已有基础输入框、聊天区、状态栏、Todo 面板、权限对话框和流式事件桥接，但输入模型仍偏“单行命令框”，渲染模型仍偏“纯文本输出框”。这使得实际使用中的三类高频场景体验较差：一是粘贴多行代码和编辑较长 prompt；二是浏览历史输入并重复修改；三是阅读包含 Markdown 结构的 assistant 输出。

本轮变更跨越 `tui`、`cli` 与本地持久化边界：输入系统需要从单字符串编辑扩展到多行缓冲与历史管理；Markdown 渲染需要让 UI 对消息内容做结构化分段；流式取消则需要从 TUI 向 QueryLoop/后台任务建立安全的中断通道。此外，迭代 15 已经引入 thinking 流式显示与折叠摘要，本轮要在此基础上补齐“展开查看”的交互闭环。

## Goals / Non-Goals

**Goals:**
- 让 TUI 输入框支持多行输入、代码粘贴、历史浏览与更完整的编辑快捷键
- 为常见 Markdown 元素提供稳定、可读、终端友好的基础渲染
- 为流式输出提供显式取消与清屏等常用交互控制
- 将 thinking 块从“只展示状态”扩展为“可折叠/展开查看”的交互对象
- 为输入历史增加本地持久化，使重启后仍可复用近期 prompt

**Non-Goals:**
- 不实现完整富文本排版引擎或像浏览器一样的 Markdown/CSS 语义
- 不在本轮引入鼠标交互、复杂窗口布局重构或可点击 UI 控件
- 不把历史系统扩展为全文搜索、标签、跨会话同步等更复杂能力
- 不在本轮实现 thinking 内容的导出、复制或专门的侧边面板
- 不重构 QueryLoop 的整体运行模型，只补齐取消输出所需的最小接口

## Decisions

### D1: 输入模型从单行字符串升级为多行缓冲

**选择**: 在 `tui::App` 中引入按行管理的输入缓冲（逻辑上可表示为 `Vec<String>` 或带光标坐标的文本缓冲结构），保留现有键盘事件入口，但让 Enter/Shift+Enter 分别承担提交与换行语义。

**理由**: 多行输入、代码粘贴、行内/按词移动、本地编辑都需要二维光标模型。继续在单个字符串上打补丁会让快捷键、渲染和粘贴处理彼此耦合。

**替代方案**: 继续使用单字符串并通过 `\n` 模拟多行 —— 拒绝，因为这会让光标定位、历史回放与可视化换行都变得脆弱。

### D2: 历史持久化采用轻量顺序日志

**选择**: 将输入历史持久化到 `~/.config/rust-claude-code/history`，采用简单逐条追加/覆盖的文本或 JSONL 结构；启动时加载到内存，提交 prompt 后去重追加，Up/Down 在内存中导航。

**理由**: 历史记录只需要“可靠、可恢复、易调试”，不需要复杂索引。轻量顺序文件足够支持当前需求，并便于后续扩展。

**替代方案**: 使用 sqlite 或 sled —— 拒绝，因为超出当前问题规模，会引入不必要依赖和迁移成本。

### D3: Markdown 渲染采用“预解析为块 + ratatui 渲染 span/line”

**选择**: 在 UI 渲染前先把 assistant 文本解析成有限集合的块类型（段落、标题、列表、代码块、行内代码增强段），再映射到 ratatui 的 `Line`/`Span` 结构。代码块语法高亮作为可选增强，优先保证边框和基础着色可用。

**理由**: 直接在渲染阶段边扫边画会让滚动、换行和样式逻辑纠缠。预解析可让渲染层保持可测试，也能在没有高亮依赖时优雅降级。

**替代方案**: 完整接入重量级 Markdown 渲染器 —— 拒绝，因为终端 UI 场景只需要受控子集，且需要与现有消息布局紧密集成。

### D4: 流式取消通过显式 cancellation channel 传递

**选择**: 从 TUI 到后台 worker 增加取消信号通道；当用户按 `Escape` 或空输入状态下的 `Ctrl+C` 触发取消时，只中断当前 streaming 请求/展示，不销毁整个 TUI。真正退出应用仍由“空闲且无流式任务时的 Ctrl+C”负责。

**理由**: 用户需要区分“停止这一轮输出”和“退出程序”。通过取消通道可把语义边界说清楚，也避免粗暴杀掉任务造成终端状态不一致。

**替代方案**: 直接 abort 后台任务句柄 —— 可作为实现细节，但不应作为外部行为契约；仍需有显式取消语义和状态同步。

### D5: Thinking 展开状态保存在 UI 层，而不是消息内容层

**选择**: 保持 `ContentBlock::Thinking` 仍只承载内容与元数据；展开/折叠状态由 `App` 维护（按消息索引和 block 索引跟踪），默认折叠，用户通过快捷键切换当前选中 thinking 块的展开状态。

**理由**: 展开是纯显示状态，不应污染持久化消息模型。放在 UI 层更符合单向职责，也避免 session 文件混入瞬时展示信息。

**替代方案**: 在消息模型里加 `expanded` 字段 —— 拒绝，因为这不是领域数据，而是瞬时视图状态。

### D6: 修改 `extended-thinking` 而不是新增独立 spec

**选择**: thinking 的“可展开查看”是现有 capability 的行为增强，因此以 `extended-thinking` 的 `MODIFIED Requirements` 表达；输入、Markdown、流式控制则各自作为新增 capability。

**理由**: 这样能保持 spec 语义连续，不把同一条能力拆成多个彼此重叠的 spec。

## Risks / Trade-offs

- **[输入缓冲复杂度上升]** → 用清晰的数据结构封装光标、行缓冲与编辑操作，并优先补单元测试覆盖粘贴、回车、历史切换等高频路径。
- **[终端渲染性能波动]** → Markdown 只支持受控子集，代码块高亮允许降级为纯边框+单色样式，避免每帧做昂贵解析。
- **[取消语义与退出语义冲突]** → 明确约束：有流式任务时 `Escape`/`Ctrl+C` 先取消本轮；仅在空闲态下 `Ctrl+C` 才退出。
- **[历史文件损坏或兼容问题]** → 采用宽松读取策略，单条坏记录跳过；写入路径使用原子覆盖或安全追加，避免阻断启动。
- **[thinking 展开交互缺少鼠标支持]** → 本轮只保证键盘可达；交互入口在 help/状态提示中说明，后续如有需要再加鼠标。
