## Context

当前 `crates/tui/src/ui.rs` 已经把聊天内容渲染为 `Paragraph` 并应用了 `scroll_offset`，但 `crates/tui/src/app.rs` 还没有提供任何针对聊天视口的滚动按键处理，因此用户无法真正操作这个偏移量。与此同时，输入区已经占用了 `Up`/`Down`、`Home`/`End`、`Ctrl+A`/`Ctrl+E` 等编辑快捷键；如果直接复用这些按键来滚动聊天区，会破坏多行输入与历史浏览语义。

这个问题横跨 `app.rs` 的输入事件分发、`ui.rs` 的聊天内容高度计算与帮助文案更新。要让滚动行为稳定，需要让“聊天区到底有多少可滚动行”与“当前是否应自动追随最新消息”使用同一套计算逻辑，避免出现渲染层和状态层各自维护偏移规则而产生漂移。

## Goals / Non-Goals

**Goals:**
- 为聊天内容区域补齐明确、可发现的键盘滚动能力，允许查看超出当前视口的历史消息
- 在滚动查看历史与追随最新消息之间建立稳定状态模型，避免新消息到达时视口乱跳
- 保持输入编辑、历史浏览、thinking 展开与流式输出行为不被聊天滚动破坏
- 补充帮助文案与测试，让“聊天区支持滚动”成为真实能力而不是文档宣称

**Non-Goals:**
- 不在本轮引入鼠标滚轮、触控板手势或复杂可视滚动条
- 不重构整体布局，也不改变聊天消息的渲染样式
- 不新增全文搜索、消息定位或按消息级折叠等高级浏览能力
- 不改变输入区现有 Enter/Shift+Enter、Up/Down、Home/End 的编辑语义

## Decisions

### D1: 使用专用聊天导航快捷键，而不是复用输入编辑键

**选择**: 使用 `PageUp` / `PageDown` 执行按页滚动，使用 `Ctrl+Home` / `Ctrl+End` 跳转到最旧/最新可见位置；保留 `Up` / `Down`、`Home` / `End` 继续服务于输入框光标和历史导航。

**理由**: 当前输入区已经依赖这些基础方向键实现多行编辑与历史浏览。若让聊天滚动抢占同一组按键，会让用户在编辑 prompt 时丢失稳定预期。专用导航键冲突最小，也更符合终端应用对长内容翻页的常见约定。

**替代方案**: 复用 `Up` / `Down` 在输入为空时滚动聊天区 —— 拒绝，因为这会让“空输入”和“有草稿”两种状态的按键含义切换过于隐蔽，也会让历史浏览行为变得不连续。

### D2: 保留顶部偏移量模型，并显式增加 `follow_output` 状态

**选择**: 继续使用 `scroll_offset` 表示聊天段落从顶部起始的可见偏移，同时新增布尔状态（如 `follow_output`）表示视口是否应自动保持在底部。用户主动执行向上滚动后关闭自动追随；当滚动回到底部或执行跳到底部后重新开启。

**理由**: 现有 UI 已经直接把 `scroll_offset` 传给 `Paragraph::scroll`。保留这个模型可以减少重构范围，而 `follow_output` 则能清晰表达“是用户在看历史，还是应该追随最新输出”的语义边界。

**替代方案**: 改为存储“距离底部的反向偏移” —— 拒绝，因为这会让现有渲染接口和边界计算全部倒置，收益不足。

### D3: 抽取共享的聊天行构建/计数逻辑用于滚动边界计算

**选择**: 把聊天区的消息渲染前处理抽成共享辅助逻辑，由它同时产出 `Vec<Line>` 与总行数，供 `ui.rs` 渲染和 `app.rs` 计算最大滚动范围共同使用。

**理由**: 如果 `app.rs` 自己估算消息高度、`ui.rs` 自己再次生成显示内容，两边很容易因为 Markdown、thinking 展开、工具结果截断等细节不一致而出现“滚不到底”或“滚过头”的错误。

**替代方案**: 在按键处理时用粗略估算（例如消息数乘固定高度） —— 拒绝，因为 assistant 长文本、代码块和 thinking 展开都会让高度远超固定估值。

### D4: 新消息到达时按“是否在底部”决定视口行为

**选择**: 当 `follow_output` 为 true 时，流式增量、完整 assistant 消息、tool/thinking 事件到达后自动把视口移动到最新内容；当用户已滚动离开底部时，新增内容只更新缓冲，不改变当前可见窗口。

**理由**: 终端聊天体验里，用户查看历史时最怕被新 token 强制拉回底部；而在正常对话时又期望实时看到最新输出。将两种行为显式区分最符合直觉。

**替代方案**: 始终自动跳到底部 —— 拒绝，因为无法阅读历史；始终保持固定偏移 —— 也拒绝，因为正常对话时会看不到正在生成的新内容。

## Risks / Trade-offs

- **[聊天总行数需要实时计算]** → 通过共享辅助函数统一渲染前展开逻辑，避免状态层和渲染层重复实现
- **[不同终端对 PageUp/PageDown 支持不完全一致]** → 保留 `Ctrl+Home` / `Ctrl+End` 作为额外导航入口，并在帮助文案中明确说明
- **[流式输出期间滚动边界频繁变化]** → 每次消息/流更新后重新基于最新总行数做 clamp，保证偏移始终落在合法范围
- **[长消息渲染行数与可视高度计算不一致]** → 使用与最终渲染相同的聊天行构建结果来计算最大偏移，避免近似值带来的 off-by-one 问题
